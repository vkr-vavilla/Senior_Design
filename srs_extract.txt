1 PRODUCT CONCEPT
Final Round is being built as a single platform for software engineering interview practice. Our goal
is to give candidates one place where they can work through technical, resume-based, and behavioral
interview rounds instead of jumping across several different tools.
1.1 PURPOSE AND USE
The product is meant to simulate a realistic software engineering interview through a web browser. A
user signs in, selects a target role or company-style interview, uploads a resume, and starts a session.
From there, the system moves through coding prompts, resume follow-up questions, and behavioral
questions in one continuous flow.
We are building Final Round as an active practice tool rather than a passive study resource. Instead
of only showing sample questions, the system reacts to what the user writes, keeps the interview moving,
and gives structured feedback at the end of the session. The current version focuses on a text-first inter￾view flow because it is easier for us to test, evaluate, and demonstrate reliably on local infrastructure.
Voice interaction is planned for a later stage once the text pipeline is stable.
1.2 INTENDED AUDIENCE
The main audience for Final Round is undergraduate computer science students and early-career can￾didates preparing for internships, new graduate roles, and entry-level software engineering interviews.
These users need something affordable, available on demand, and realistic enough to show where they
are weak before they sit in an actual interview.
Secondary users include faculty reviewers, senior design staff, and university career support groups
that may want to use the platform as a demo system or as a scalable interview-practice tool. The
product is not intended to replace a live interviewer or make hiring decisions. Its purpose is to help
users practice, reflect on their performance, and improve across repeated sessions.

2 PRODUCT DESCRIPTION
Final Round combines the main parts of software engineering interview preparation into one platform.
The system is designed to take a user from setup, to interview, to feedback without forcing them to
switch between unrelated tools.
2.1 FEATURES & FUNCTIONS
In the current scope, Final Round is expected to provide the following main capabilities:
• user sign-in and session management through a browser-based interface;
• target role or company-style selection before the interview starts;
• local resume upload and parsing so the interviewer can ask grounded follow-up questions;
• a structured interview flow with coding, resume, and behavioral sections;
• feedback that summarizes strengths, weak points, and practical next steps; and
• a recent-session view so users can revisit earlier practice attempts.
The platform is intended for interview practice. It does not make hiring decisions, replace employer
screening systems, or guarantee a user will perform well in a real interview. When code execution is
supported, it will run inside a controlled environment. When full execution is not available, the system
will still evaluate the logic, structure, and completeness of the submitted answer.

6 SAFETY REQUIREMENTS
Even though Final Round is a software platform, it still needs safety-oriented controls because it pro￾cesses uploaded files, accepts code-like input, and runs on shared institutional hardware.
6.1 CONTROLLED CODE EXECUTION ENVIRONMENT
6.1.1 DESCRIPTION
Any feature that executes or evaluates user-submitted code shall do so in a controlled environment
with explicit resource limits, restricted permissions, and no unnecessary access to host services. If a
safe execution path is unavailable, the platform shall fall back to static or logical analysis rather than
running untrusted code directly on the host.
6.1.2 SOURCE
Project charter system overview and the need to evaluate coding responses safely.
6.1.3 CONSTRAINTS
Strict isolation can reduce execution fidelity and may limit supported languages or available libraries.
The implementation must prefer safety over feature breadth.
6.1.4 STANDARDS
Safe execution practices should align with secure software development guidance such as NIST SSDF
and internal lab computing rules [3].
6.1.5 PRIORITY
Critical
6.2 GRACEFUL FAILURE AND FALLBACK HANDLING
6.2.1 DESCRIPTION
If a dependent service fails, such as resume parsing, model inference, or local speech components, the
system shall fail in a controlled and user-readable way. Error states shall not expose raw stack traces to
normal users, and the platform shall prefer a degraded but usable mode whenever feasible.
6.2.2 SOURCE
Sprint 4 planning for fallback behavior and practical demo needs.
6.2.3 CONSTRAINTS
Graceful degradation requires extra state handling and testing effort. Not every component can offer a
meaningful fallback, but the user should still receive a clear message about what happened.
6.2.4 STANDARDS
Operational failures should be logged for maintainers while user-facing messages remain concise and
non-sensitive.
6.2.5 PRIORITY
High
6.3 UPLOAD VALIDATION AND RESOURCE LIMITS
6.3.1 DESCRIPTION
The system shall validate uploaded resumes and other user-supplied artifacts before processing them.
Allowed file types, file sizes, and parsing paths shall be restricted so malformed or oversized files do not

6.3.4 STANDARDS

7 SECURITY REQUIREMENTS
Because Final Round stores resumes, interview transcripts, and evaluation reports, the platform must
protect user data and prevent misuse of both the application and the local infrastructure it runs on.
7.1 AUTHENTICATED ACCESS AND SESSION CONTROL
7.1.1 DESCRIPTION
Protected routes in the platform shall require user authentication. Session tokens or equivalent creden￾tials shall expire after a defined period of inactivity, and a user shall not be able to access another user’s
saved interviews or reports through simple URL changes or client-side manipulation.
7.1.2 SOURCE
Basic platform security expectations and user account design.
7.1.3 CONSTRAINTS
The prototype should remain easy to demo, but convenience cannot come from disabling access control
on stored user data.
7.1.4 STANDARDS
Authentication and access control behavior should align with OWASP ASVS guidance for session man￾agement and authorization [2].
7.1.5 PRIORITY
Critical
7.2 PROTECTED TRANSPORT AND CREDENTIAL STORAGE
7.2.1 DESCRIPTION
Credentials shall never be stored in plaintext. Passwords or equivalent secrets shall be protected using a
modern password hashing approach, and network transport for deployed instances shall use encrypted
channels whenever the system is accessed over a network.
7.2.2 SOURCE
Standard software security practice and sponsor expectations.
7.2.3 CONSTRAINTS
Local development environments may use self-signed certificates or isolated lab networking during
setup, but the deployed demonstration environment must still handle secrets responsibly.
7.2.4 STANDARDS
Encrypted transport should follow TLS 1.3 guidance [4], and password hashing should follow modern
memory-hard recommendations such as Argon2 usage guidance [5].
7.2.5 PRIORITY
Critical
7.3 DATA PROTECTION AND RETENTION CONTROL
7.3.1 DESCRIPTION
Resumes, transcripts, code submissions, and feedback reports shall be stored only as long as needed
for project use, debugging, or user review. Administrative access to stored data shall be limited, and

7.3.2 SOURCE
Project charter constraints related to privacy and institutional infrastructure.
7.3.3 CONSTRAINTS

7.3.4 STANDARDS
Storage and access policies should follow least-privilege principles and secure software development
