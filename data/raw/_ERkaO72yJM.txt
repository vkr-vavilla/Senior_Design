For today's software engineering interview, please explain torrenting. Unlike a traditional client server setup where you, the client, downloads a complete file from some server. Torrenting uses a peer-to-peer network. The file you want to download is first split into many different pieces. Peers that are participating in the network might have some pieces of this file or they might have the complete file. People that have the complete file are known as ceders and they can help you get some of that file. And you also have peers or leachers who are currently downloading a file. When you want to download a file, you're not just downloading the entire file at once. You're downloading pieces of the file from people all across the network, but you're also uploading pieces of the file that you have to other people that are trying to download this file. This way, a decentralized network of people can efficiently distribute files to each other without the need for a centralized server. Okay? So, everybody participating in the peer-to-peer network is both downloading and uploading different parts of the file at the same time until everybody has the complete file. Now, how do you find out who has what files or join one of these peer-to-peer networks? A fairly common way of doing this is called a distributed hash table. Each node that's participating in this torrent network stores some information about what nodes have what torrent. In this simple example, let's say I want to download all of One Piece. My client would reach out to Pier 1. Pier 1 doesn't have One Piece, but it knows that Pier 3 does. So, I can go to Pier 3, which can grant me access to the swarm, where I can download and upload the file as usual.
